# veda_linux_project2 개발 문서

## 1. 프로젝트 개요

이 프로젝트는 TCP 소켓 기반 장치 제어 시스템입니다.

서버는 데몬 프로세스로 동작하며 LED, BUZZER, LIGHT SENSOR, FND 장치를 제어합니다. 각 장치 제어 코드는 동적 라이브러리(`.so`)로 분리되어 있고, 서버는 실행 중 `dlopen()`과 `dlsym()`을 사용하여 장치 제어 함수를 로드합니다.

클라이언트는 메뉴 기반 CLI 프로그램이며 TCP 5100 포트로 서버에 명령을 전송합니다. 사용자는 번호 입력 방식으로 LED, BUZZER, LIGHT SENSOR, FND를 제어할 수 있습니다.

추가 기능으로 C 기반 웹 상태 서버를 제공합니다. 장치 제어 서버는 현재 상태를 `exec/status.txt`에 기록하고, 웹 상태 서버는 이 파일을 읽어 브라우저에서 현재 장치 상태를 확인할 수 있도록 HTML 응답을 생성합니다.

## 3. 세부 구현 내용

### 서버

- TCP 5100 포트에서 클라이언트 접속을 대기합니다.
- 클라이언트가 보낸 문자열 명령을 해석하여 장치 제어 함수로 연결합니다.
- `-d` 옵션이 주어지면 `fork()`와 `setsid()`를 사용하여 데몬 프로세스로 동작합니다.
- 장치 상태가 변경될 때마다 `exec/status.txt`를 갱신합니다.

### 클라이언트

- 메뉴 기반 CLI를 제공합니다.
- 사용자가 선택한 메뉴를 서버 명령 문자열로 변환하여 TCP socket으로 전송합니다.
- 서버 응답을 화면에 출력합니다.
- FND countdown 기능은 `fork()`로 별도 프로세스를 만들어 처리합니다.
- 종료 시 LED와 BUZZER를 off하는 명령을 전송하여 장치가 켜진 채로 남지 않도록 했습니다.

### 동적 라이브러리

- LED, BUZZER, LIGHT, FND 제어 코드는 각각 공유 라이브러리로 빌드됩니다.
- 서버는 `dlopen()`으로 라이브러리를 로드하고, `dlsym()`으로 필요한 함수를 찾아 호출합니다.
- 장치별 구현을 서버 본체와 분리하여 기능 추가와 유지보수가 쉽도록 구성했습니다.

### 조도센서 자동 제어

- 클라이언트에서 `LIGHT SENSOR ON`을 선택하면 서버가 현재 센서 값을 한 번 읽어 응답합니다.
- 이후 서버 내부 thread가 0.5초마다 조도센서 값을 읽습니다.
- 빛이 감지되지 않으면 LED를 켜고, 빛이 감지되면 LED를 끄도록 자동 제어합니다.
- `LIGHT SENSOR OFF` 명령을 받으면 센서 thread를 종료합니다.

### 웹 상태 서버

- `web_status`는 C socket API만 사용하여 HTTP 요청을 처리합니다.
- 장치 제어 서버와 직접 socket 통신하지 않고 `exec/status.txt`를 읽어 현재 상태를 표시합니다.
- HTML 응답을 동적으로 생성하여 client 접속 상태, LED, 밝기, BUZZER, SENSOR, LIGHT, FND, 갱신 시간을 보여줍니다.
- 브라우저 화면은 0.3초마다 새로고침되어 최신 상태를 확인할 수 있습니다.

### 빌드 구조

- 루트 `Makefile`에서 서버, 클라이언트, 웹 상태 서버를 빌드합니다.
- 서버 실행 파일과 클라이언트 실행 파일은 `exec/`에 생성됩니다.
- 장치 공유 라이브러리는 `exec/lib/`에 생성됩니다.
- 클라이언트 장비에서 wiringPi가 없어도 `make client`만 실행하면 클라이언트 프로그램을 빌드할 수 있습니다.

## 4. 문제점 및 보완 사항

### 동적 라이브러리 실행 경로 문제

서버는 장치 공유 라이브러리를 `./lib/lib*.so`와 `./exec/lib/lib*.so` 경로에서 순서대로 로드합니다. 따라서 프로젝트 루트에서는 `./exec/server -d`로 실행할 수 있고, `exec` 디렉터리에서는 `./server -d`로 실행할 수 있습니다.

보완:

- 서버가 두 라이브러리 경로를 모두 시도하도록 구현했습니다.
- 빌드 산출물을 `exec/` 아래에 모아 서버 실행 위치와 라이브러리 위치가 일관되도록 정리했습니다.

### 클라이언트 장비의 wiringPi 의존성 문제

서버용 장치 라이브러리는 wiringPi를 사용하므로, 일반 우분투 클라이언트 장비에서 전체 `make`를 실행하면 `wiringPi.h`가 없어 빌드가 실패할 수 있습니다.

보완:

- `make server`, `make client`, `make web` 타깃을 분리했습니다.
- 클라이언트 장비에서는 `make client`만 실행하면 wiringPi 없이 클라이언트 실행 파일을 빌드할 수 있습니다.

### 센서 출력으로 인한 클라이언트 메뉴 가독성 문제

초기 구조에서는 조도센서 thread가 0.5초마다 client 화면에 값을 계속 출력할 수 있어 메뉴 입력 화면이 깨질 수 있었습니다.

보완:

- `LIGHT SENSOR ON` 선택 시 client에는 현재 조도센서 값을 한 번만 출력하도록 변경했습니다.
- 이후 서버 내부 thread가 센서 값을 주기적으로 읽고 LED를 자동 제어합니다.
- 지속적인 센서 상태 확인은 `exec/status.txt`와 웹 상태 대시보드에서 확인하도록 역할을 분리했습니다.

### 서버의 단일 클라이언트 처리 구조

현재 서버는 한 번에 하나의 클라이언트 연결을 처리하는 구조입니다. 여러 클라이언트가 동시에 접속하는 환경에서는 다음 클라이언트 처리가 지연될 수 있습니다.

보완 방향:

- 추후 `pthread_create()`를 사용하여 client 연결별 thread를 생성하면 다중 클라이언트 처리가 가능합니다.
- 또는 `select()`/`epoll()` 기반 구조로 확장할 수 있습니다.
- 다중 클라이언트 구조로 확장할 경우 실제 장치 제어 함수 호출에는 mutex를 적용하여 명령 충돌을 방지해야 합니다.

### 상태 공유 방식의 한계

웹 상태 서버는 `exec/status.txt` 파일을 읽어 상태를 표시합니다. 구현이 단순하고 디버깅이 쉽지만, 엄밀한 동기화가 필요한 대규모 시스템에는 적합하지 않을 수 있습니다.

보완 방향:

- 추후 System V shared memory 또는 POSIX shared memory를 사용하면 서버와 웹 상태 서버 간 상태 공유를 더 직접적으로 처리할 수 있습니다.
- 현재 과제 범위에서는 파일 기반 상태 공유가 구현 난이도와 안정성 측면에서 적합하다고 판단했습니다.
